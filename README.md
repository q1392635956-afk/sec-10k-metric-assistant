# SEC 10-K Metric Assistant

**Cited financial metric computation from Apple's FY2025 annual filing.**

A GenAI pipeline that retrieves evidence from a 10-K filing, extracts numbers with an LLM, computes derived financial metrics with Python, and returns an auditable, cited answer — compared against a direct-LLM baseline.

---

## Quick start

```bash
# 1. Clone and enter the project
git clone <repo-url>
cd sec-10k-metric-assistant

# 2. Install dependencies (Python 3.10+ required)
pip install -r requirements.txt

# 3. Set your OpenAI API key
cp .env.example .env
# Edit .env and add: OPENAI_API_KEY=sk-...

# 4. Add the 10-K text (see data/README.md for how to obtain it)
#    Place the file at:
#    data/apple_2025_10k.txt

# 5. Run the Streamlit app
streamlit run app.py
```

The first run builds the embedding index (this takes 1–3 minutes and costs a small amount in API credits). Subsequent runs load from cache instantly.

## Project context

### User and workflow

**User:** Financial analyst reviewing Apple's FY2025 annual report.

**Workflow:** The analyst asks a question about a derived financial metric. The system locates the relevant passages in the 10-K, extracts the raw numbers with an LLM, computes the metric using verified Python arithmetic, and returns a cited explanation. The analyst can then verify the source passages before using the result.

### Why GenAI is useful here

Annual 10-K filings are long (100–200 pages) and the specific numbers needed for a derived metric may be spread across different sections. GenAI is used for three language tasks where it genuinely helps:

1. **Routing** — understanding which metric the analyst is asking about from a natural language question
2. **Extraction** — reading a table or paragraph and identifying the correct numeric value for a specific line item
3. **Explanation** — writing a concise, professional summary of the computed result with source attribution

The actual math (division, subtraction, percentage) is **never delegated to the LLM**. Python computes it deterministically.

### What was compared

| System | Retrieval | Formula | Python math | Cited source |
|--------|-----------|---------|-------------|--------------|
| Metric-aware (this project) | Yes | Yes | Yes | Yes |
| Baseline | No | No | No | No |

The baseline asks the same question directly to `gpt-4o-mini` with no grounding. It often gives plausible but unverifiable numbers or declines to answer for recent filings it wasn't trained on.

## Supported metrics

| Key | Name | Formula |
|-----|------|---------|
| `gross_margin` | Gross Margin | `(gross_profit / revenue) × 100` |
| `operating_margin` | Operating Margin | `(operating_income / revenue) × 100` |
| `net_profit_margin` | Net Profit Margin | `(net_income / revenue) × 100` |
| `current_ratio` | Current Ratio | `current_assets / current_liabilities` |
| `rd_growth` | R&D Expense YoY Growth | `((rd_current − rd_prior) / rd_prior) × 100` |

Metric definitions and required fields live in `metrics.json`.

## System design

```
User question
    │
    ▼
llm_utils.classify_metric()        ← LLM: routes question to a metric key
    │
    ▼
retriever.retrieve()               ← Embeddings search over 10-K chunks
    │
    ▼
llm_utils.extract_values()         ← LLM: pulls numbers from evidence text
    │
    ▼
metric_engine.compute_metric()     ← Python: deterministic arithmetic
    │
    ▼
llm_utils.format_answer()          ← LLM: writes cited explanation
    │
    ▼
Streamlit UI display
```

### Key design decisions

**Token-based chunking (ingest.py):** Uses `tiktoken` with 800-token chunks and 100-token overlap. Token-based splitting keeps chunks within the embedding model's context limit and the overlap prevents answers from being split across chunk boundaries.

**Embedding cache (retriever.py):** Embeddings are computed once and saved to `data/embeddings_cache.pkl`. This avoids redundant API calls across sessions. Delete the cache file to rebuild after updating the 10-K text.

**Retrieval query augmentation:** The retrieval query combines the user's question with the metric name and required field names (e.g., "gross_profit", "revenue"). This improves recall for financial table content that may not use the same phrasing as the question.

**Structured JSON extraction (llm_utils.py):** The `extract_values` function uses `response_format={"type": "json_object"}` so the model always returns parseable JSON. Field-specific hints in the prompt (e.g., "Total current assets from the balance sheet") reduce extraction errors.

**No LLM math (metric_engine.py):** Each metric has an explicit Python function. This eliminates arithmetic hallucination and makes the computation fully auditable. The formula in `metrics.json` is displayed to the user alongside the result.

## Running the evaluation

```bash
python run_evaluation.py
```

This runs 10 questions from `eval/test_questions.csv` through both systems and saves results to `eval/results.csv`. The console prints:

- Metric classification accuracy (how often the LLM correctly identified which metric was requested)
- Computation success rate (how often all required values were extracted)

After running, fill in `eval/evaluation_summary.md` with your qualitative analysis.

### What the evaluation measures

| Dimension | How measured |
|-----------|-------------|
| Metric classification accuracy | `metric_key == expected_metric` |
| Computation success rate | `computed_value is not None` |
| Numeric accuracy | Manual comparison of `computed_value` vs `expected_value` |
| Answer quality | Qualitative review of `answer` column |

## Limitations

- **Scope:** One company (Apple), one filing (FY2025), five metrics. Extending to other companies or metrics requires adding data and updating `metrics.json`.
- **Extraction errors:** The LLM may extract the wrong row from a financial table, especially when multiple years are presented side by side. Always verify extracted values against the source passages.
- **Retrieval miss:** If the relevant passage is not in the top-k retrieved chunks, the extraction step will fail. Increasing `top_k` in `retriever.py` trades cost for recall.
- **FY2025 data:** The model's training data may not include Apple's FY2025 10-K (filed late 2025). The metric-aware system is grounded in the actual filing; the baseline may give outdated or uncertain answers for this reason.
- **Table parsing:** Plain-text 10-K files can have inconsistent formatting in financial tables. If numbers are not extracted correctly, try a cleaner text extraction of the source PDF.

---

## Where a human should stay involved

1. **Verify extracted values** — check that `gross_profit`, `revenue`, etc. in the "Computation Details" panel match the actual 10-K line items before using the result.
2. **Interpret edge cases** — metrics like the current ratio require understanding of Apple's specific balance sheet structure (e.g., how commercial paper and marketable securities are classified).
3. **Approve for external use** — any metric used in a report, model, or decision should be signed off by a human analyst who has reviewed the source filing.
4. **Update for new filings** — when a new 10-K is released, the cache must be rebuilt and expected values in the test set must be updated manually.
