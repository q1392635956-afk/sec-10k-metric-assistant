# SEC 10-K Metric Assistant

**Cited financial metric computation from Apple's FY2025 annual filing.**

A GenAI pipeline that retrieves evidence from a 10-K filing, extracts numbers with an LLM, computes derived financial metrics with Python, and returns an auditable, cited answer вЂ” compared against a direct-LLM baseline.

---

## Quick start

```bash
# 1. Clone and enter the project
git clone <repo-url>
cd sec-10k-metric-assistant

# 2. Install dependencies (Python 3.10+ required)
pip install -r requirements.txt

# 3. Set your Gemini API key
cp .env.example .env
# Edit .env and add your key:
#   GEMINI_API_KEY=your_google_ai_studio_api_key_here
# Get a free key at https://aistudio.google.com/app/apikey

# 4. Add the 10-K text (see data/README.md for how to obtain it)
#    Place the file at:
#    data/apple_2025_10k.txt

# 5. Run the Streamlit app
streamlit run app.py
```

The first run builds the TF-IDF index (under one second) and caches it to `data/tfidf_cache.pkl`. Subsequent runs load from cache instantly. No API calls are needed for retrieval.

## Project context

### User and workflow

**User:** Financial analyst reviewing Apple's FY2025 annual report.

**Workflow:** The analyst asks a question about a derived financial metric. The system locates the relevant passages in the 10-K, extracts the raw numbers with an LLM, computes the metric using verified Python arithmetic, and returns a cited explanation. The analyst can then verify the source passages before using the result.

### Why GenAI is useful here

Annual 10-K filings are long (100вЂ“200 pages) and the specific numbers needed for a derived metric may be spread across different sections. GenAI is used for three language tasks where it genuinely helps:

1. **Routing** вЂ” understanding which metric the analyst is asking about from a natural language question
2. **Extraction** вЂ” reading a table or paragraph and identifying the correct numeric value for a specific line item
3. **Explanation** вЂ” writing a concise, professional summary of the computed result with source attribution

The actual math (division, subtraction, percentage) is **never delegated to the LLM**. Python computes it deterministically.

### What was compared

| System | Retrieval | Formula | Python math | Cited source |
|--------|-----------|---------|-------------|--------------|
| Metric-aware (this project) | Yes | Yes | Yes | Yes |
| Baseline | No | No | No | No |

The baseline asks the same question directly to Gemini with no grounding. It often gives plausible but unverifiable numbers or declines to answer for recent filings it was not trained on.

### Key design decisions

**Character-based chunking (ingest.py):** Uses 1500-character chunks with 150-character overlap. Chunks are broken at newline boundaries so financial table rows stay intact. This size keeps each major financial statement (income statement, balance sheet) in its own dedicated chunk, which improves TF-IDF retrieval precision.

**TF-IDF retrieval (retriever.py):** Uses scikit-learn's `TfidfVectorizer` with unigrams and bigrams. The index is built from the 10-K chunks once and cached to `data/tfidf_cache.pkl`. No API calls are needed for retrieval. Each metric in `metrics.json` includes `search_terms` вЂ” exact phrases from the 10-K document вЂ” that are appended to the retrieval query to improve recall for financial table content.

**Structured JSON extraction (llm_utils.py):** The `extract_values` function uses `response_mime_type="application/json"` so the model always returns parseable JSON. Field-specific hints in the prompt (e.g., "Total current assets from the balance sheet as of September 27, 2025") reduce extraction errors.

**No LLM math (metric_engine.py):** Each metric has an explicit Python function. This eliminates arithmetic hallucination and makes the computation fully auditable. The formula in `metrics.json` is displayed to the user alongside the result.

**Thinking disabled (llm_utils.py):** All Gemini calls set `thinking_budget=0`. The three LLM tasks here (classification, extraction, explanation) are straightforward language tasks that do not benefit from chain-of-thought reasoning. Disabling thinking avoids token-budget exhaustion that would cause `response.text` to be `None`.

## Running the evaluation

```bash
python run_evaluation.py
```

This runs 10 questions from `eval/test_questions.csv` through both systems and saves results to `eval/results.csv`. The console prints:

- Metric classification accuracy (how often the LLM correctly identified which metric was requested)
- Computation success rate (how often all required values were extracted)

After running, fill in `eval/evaluation_summary.md` with your qualitative analysis.

### What the evaluation measures

| Dimension | How measured |
|-----------|-------------|
| Metric classification accuracy | `metric_key == expected_metric` |
| Computation success rate | `computed_value is not None` |
| Numeric accuracy | Manual comparison of `computed_value` vs `expected_value` |
| Answer quality | Qualitative review of `answer` column |

|----------|---------|-------------|
| `GEMINI_API_KEY` | *(required)* | Your Google AI Studio API key |

Set variables in `.env` (local) or your shell environment (CI/production).

### Why `gemini-2.5-flash-lite` is the recommended model

Flash-Lite is sufficient for all three LLM tasks in this project:

- **Classification** is a one-word routing decision from a fixed list of five options.
- **Extraction** reads a short financial table and maps line items to a JSON schema вЂ” a structured reading task, not reasoning.
- **Explanation** writes 3вЂ“4 sentences summarizing a pre-computed number вЂ” the math is already done by Python.

Because Python handles all arithmetic, the LLM never needs to reason about calculations. Flash-Lite provides fast, reliable outputs for these narrow language tasks at lower cost than larger models.

## Limitations

- **Scope:** One company (Apple), one filing (FY2025), five metrics. Extending to other companies or metrics requires adding data and updating `metrics.json`.
- **Extraction errors:** The LLM may extract the wrong row from a financial table, especially when multiple years are presented side by side. Always verify extracted values against the source passages.
- **Retrieval miss:** If the relevant passage is not in the top-k retrieved chunks, the extraction step will fail. Increasing `top_k` in `retriever.py` trades latency for recall.
- **FY2025 data:** The model's training data may not include Apple's FY2025 10-K (filed late 2025). The metric-aware system is grounded in the actual filing; the baseline may give outdated or uncertain answers for this reason.
- **Table formatting:** The 10-K is converted from iXBRL HTML to plain text. Numbers in complex multi-column tables may occasionally appear on separate lines. If extraction fails, check `data/apple_2025_10k.txt` for the relevant table.

---

## Where a human should stay involved

1. **Verify extracted values** вЂ” check that `gross_profit`, `revenue`, etc. in the "Computation Details" panel match the actual 10-K line items before using the result.
2. **Interpret edge cases** вЂ” metrics like the current ratio require understanding of Apple's specific balance sheet structure (e.g., how commercial paper and marketable securities are classified).
3. **Approve for external use** вЂ” any metric used in a report, model, or decision should be signed off by a human analyst who has reviewed the source filing.
4. **Update for new filings** вЂ” when a new 10-K is released, delete `data/tfidf_cache.pkl` to force a rebuild, and update the expected values in `eval/test_questions.csv`.
